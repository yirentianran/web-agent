# 数据库迁移分析 — Web Agent

**日期**: 2026-04-16
**范围**: 评估基于文件的存储方式是否需要迁移到数据库
**当前规模**: ~2 用户, ~17 会话, ~5MB 总数据

---

## 1. 当前存储架构

### 1.1 数据结构

| 数据类型 | 存储方式 | 格式 | 路径 |
|-----------|---------|------|------|
| 会话消息 | JSONL 文件 | 每行一个 JSON | `data/.msg-buffer/{session_id}.jsonl` |
| 会话元数据 | 空 JSONL + meta.json | 单一对象 | `data/users/{user}/claude-data/sessions/` |
| 用户内存 (L1) | JSON 文件 | 格式化 JSON | `data/users/{user}/memory.json` |
| Agent 笔记 (L2) | Markdown 文件 | 自由文本 | `data/users/{user}/memory/*.md` |
| 技能定义 | Markdown + JSON | SKILL.md + skill-meta.json | `data/shared-skills/` + `data/users/{user}/skills/` |
| 上传/输出文件 | 原始二进制 | 原文件 | `data/users/{user}/workspace/{uploads,outputs}/` |
| 审计日志 | JSONL 文件 | 按天按分类 | `data/logs/audit/` |
| 技能反馈 | JSONL 文件 | 每个技能一个文件 | `data/training/skill-feedback/` |
| QA 反馈 | JSONL 文件 | 每个反馈一个文件 | `data/training/qa/` |
| 任务文件 | JSON 文件 | 每个任务一个文件 | `data/users/{user}/tasks/` |

### 1.2 查询模式

| 查询 | 当前实现 | 复杂度 |
|-------|---------|--------|
| 列出会话 | `glob("*.jsonl")` + 读取每个文件首行 | O(N) 文件 I/O |
| 获取会话历史 | 内存 deque，冷缓存时 `readlines()` 全文件 | O(M) M=消息总数 |
| 获取会话状态 | 内存字典查找 | O(1) |
| 查询审计日志 | 按天读取 JSONL，Python 中过滤 | O(D) D=当日条目数 |
| 技能反馈分析 | `glob()` + 逐个 `json.loads()` | O(F) F=反馈文件数 |
| 列出任务文件 | `glob("*.json")` + 逐个读取 | O(N) |
| 列出工作区文件 | `workspace.rglob("*")` + `stat()` | O(F) |

### 1.3 已识别的问题

1. **无索引**: 所有查询都是线性文件扫描 — `list_sessions()` 每次页面加载都读取 N 个文件
2. **无原子性**: `memory.json` 的读-改-写没有文件锁 — 并发写入会丢失更新
3. **无事务**: 多步骤操作（创建会话 + 写元数据）可能留下不一致状态
4. **无分页**: `get_history()` 将整个 JSONL 文件读入内存（最大 445 条消息 = 445 次 `json.loads()`）
5. **清理不完整**: 会话清理只对硬编码用户 "default" 生效；上传/输出文件永不自动清理
6. **数据重复**: 会话数据存在于 3 个位置（`.msg-buffer/`、`claude-data/sessions/`、`.meta.json`），不一致
7. **冷缓存昂贵**: 磁盘回退时解析整个 JSONL 文件
8. **扩展天花板**: 当前 ~1 用户、~17 会话 — 数据量小，但随着用户增长会快速退化

## 2. 是否需要数据库？逐数据类型评估

### 2.1 会话消息

**查询需求**: 追加消息、列出会话、按会话检索历史、分页历史
**一致性**: 追加写入 — 天然安全
**并发性**: 每个会话单个写入者（agent 进程），多个读取者（WebSocket 消费者）
**结论**: **不需要数据库**。追加写入的文件模式是正确的。痛点在于 `list_sessions()` 和冷缓存回退，可以通过轻量索引和压缩解决。

### 2.2 用户内存 (L1)

**查询需求**: 读取完整内存、深度合并更新、替换内存
**一致性**: 读-改-写无锁 — 并发写入会丢失更新
**并发性**: 当前单用户，但 2+ 同时更新时会出问题
**结论**: **中等需求**。这是最可能先出问题的部分。文件锁（`fcntl.flock`）可以立即解决竞态条件。数据库在频繁并发更新之前是过度设计。

### 2.3 会话元数据

**查询需求**: 列出所有会话、读取/更新标题、删除会话
**一致性**: 标题更新是单次写入 — 安全。但数据在 3 个位置重复
**并发性**: 低 — 标题更新很少
**结论**: **低需求**。真正的问题是数据重复和不一致的清理逻辑，而不是并发。修复清理逻辑和统一到单一数据源比数据库更有帮助。

### 2.4 审计日志

**查询需求**: 追加条目、按分类/日期/用户/操作查询、验证哈希链
**一致性**: 追加写入 — 天然安全
**并发性**: 单个写入者（服务器进程）
**结论**: **不需要数据库**。追加写入的 JSONL 是审计日志的正确模式。哈希链防篡改在文件中工作良好。

### 2.5 上传/输出文件

**查询需求**: 列出、上传、下载、删除
**一致性**: 文件系统是数据源
**并发性**: 低 — 用户驱动的操作
**结论**: **不需要数据库**。二进制内容的正确存储介质就是文件。

### 2.6 技能反馈 & A/B 测试

**查询需求**: 追加反馈、聚合分析、加载单个技能的所有反馈
**一致性**: 每个条目一个文件 — 无读-改-写竞态
**并发性**: 低 — 技能执行后提交
**结论**: **低需求**。当前规模下全目录扫描可以忽略不计。

### 2.7 优先级排序

| 优先级 | 数据类型 | 紧急度 | 原因 |
|--------|---------|--------|------|
| 1 | 会话列表 | **立即** | `list_sessions()` 每次页面加载 O(N) 文件读取 |
| 2 | 用户内存原子性 | **近期** | 用户增长后并发写入会丢失更新 |
| 3 | 会话历史冷读取 | **近期** | 大会话 445 次 `json.loads()` 每次请求 |
| 4 | 会话清理 | **近期** | 只对 "default" 用户生效 |
| 5 | 技能反馈分析 | **稍后** | 全目录扫描但当前可忽略 |
| 6 | 审计日志查询 | **稍后** | 每日轮转保持文件小 |
| 7 | 文件列表 | **稍后** | `os.walk()` 可接受 |

## 3. 方案对比

### 方案 A：保持文件存储，修复最严重问题

**改动内容**:
- 添加轻量会话索引（单个 JSON 文件映射 session_id 到元数据）
- 给 `memory.json` 的读-改-写添加文件锁（`fcntl.flock`）
- 实现大会话的 JSONL 压缩
- 修复会话清理以覆盖所有用户
- 给 `list_sessions()` 和 `get_history()` 添加分页

**优点**:
- 零新依赖
- 无迁移风险
- 实现最快
- 保留基于文件的可调试性

**缺点**:
- 仍然没有真正的事务
- 索引文件本身需要并发保护
- 压缩是重新发明 WAL
- 扩展天花板仍然存在（只是推高了）

**成本**: 1-2 天实现
**扩展到**: ~50 用户, ~5K 会话

### 方案 B：SQLite（嵌入式数据库）

**改动内容**:
- 用 SQLite 替换 `message_buffer.py` 磁盘层（messages 表）
- 用 SQLite 替换 `memory.py`（user_memory 表）
- 用 SQLite 替换会话元数据文件（sessions 表）
- 审计日志保持文件（追加写入方式正确）
- 上传/输出保持文件

**优点**:
- 单个文件，零运维开销
- 开箱即用的 ACID 事务
- 索引解决 O(N) 扫描问题
- 分页很简单
- Python 标准库包含 `sqlite3` — 无新依赖
- 单进程最高 ~100K 写入/秒

**缺点**:
- 向无 SQL 的代码库引入 SQL
- 需要迁移（Alembic 或手动）
- 需要 WAL 模式支持并发读取
- `sqlite3` 是同步的 — 需要 `aiosqlite` 支持异步

**成本**: 3-5 天实现 + 迁移
**扩展到**: ~200 用户, ~50K 会话

### 方案 C：PostgreSQL（完整数据库）

**改动内容**:
- 所有结构化数据的完整模式迁移
- 连接池（asyncpg）
- 数据库迁移（Alembic）
- 独立服务依赖

**优点**:
- 多用户、并发工作负载的最佳选择
- 丰富的查询能力（全文搜索、JSONB）
- 水平扩展可能（读副本、连接池）
- 行业标准

**缺点**:
- 显著运维开销（部署、备份、监控）
- 当前规模过度设计
- 迁移复杂度高
- 增加故障域（数据库宕机 = 应用宕机）

**成本**: 1-2 周实现 + 持续运维
**扩展到**: 无限（企业级）

### 方案 D：混合模式（文件 + SQLite 热路径）

**改动内容**:
- 保持文件用于：审计日志、上传/输出、技能文件、Agent 笔记
- SQLite 用于：会话元数据、会话消息（热路径）、用户内存
- 不迁移冷/历史数据

**优点**:
- 最小化迁移范围
- 针对真正的痛点
- 审计日志保留防篡改特性
- 上传/输出保持文件（正确介质）

**缺点**:
- 两套存储系统需要维护
- 需要决定数据归属
- 代码库稍微复杂

**成本**: 3-4 天实现
**扩展到**: ~200 用户, ~50K 会话

## 4. 推荐方案

### 推荐：现在方案 A，后续方案 B

**理由**: 系统当前 ~2 用户、~5MB 数据。痛点确实存在但足够小，可以通过针对性修复解决，无需引入数据库。当达到以下触发条件时采用数据库：

1. **50+ 活跃用户**（并发写入冲突变得频繁）
2. **每用户 500+ 会话**（线性扫描明显变慢）
3. **每会话 10K+ 消息**（JSONL 解析时间超过 100ms）
4. **多次同时写入用户内存**（竞态条件显现）

在此之前，方案 A 是正确的选择：用最小复杂度修复最严重的问题。

### 方案 A 实施计划

#### 阶段 1：会话索引（解决痛点 #1）

**目标**: 消除 `list_sessions()` 的 O(N) 文件读取。

**改动**:
1. 创建 `data/.session-index.json` — 单个文件映射 `session_id` 到 `{user_id, created_at, title, status, cost_usd, message_count, size_bytes, last_active}`
2. 更新 `main_server.py` 的 `create_session()` 写入索引
3. 更新 `list_sessions()` 从索引读取（单次文件读取）
4. 更新 `delete_session()` 从索引删除
5. 更新 `update_session_title()` 更新索引
6. 添加索引重建脚本（启动时扫描文件，或按需）

**新文件**: `src/session_index.py`

**风险**: 低。索引可从磁盘重建。

#### 阶段 2：用户内存原子性（解决痛点 #2）

**目标**: 防止并发内存写入丢失更新。

**改动**:
1. 给 `memory.py` 的 `read()` 和 `update()` 方法添加 `fcntl.flock()`
2. 写入用排他锁，读取用共享锁
3. 添加带退避的重试逻辑处理锁竞争

**风险**: 低。文件锁是成熟方案且影响范围局部。

#### 阶段 3：JSONL 压缩（解决痛点 #3）

**目标**: 减少大会话的冷缓存读取时间。

**改动**:
1. 给 `MessageBuffer` 添加 `compact_session(session_id)` 方法
2. 压缩读取 JSONL，用 `gzip` 写入压缩 JSONL（`.jsonl.gz`）
3. 维护两种格式：热会话用 JSONL，压缩会话用压缩格式
4. 在会话关闭或清理时运行压缩

**风险**: 低。压缩是附加功能 — 现有 JSONL 文件继续工作。

#### 阶段 4：修复清理（解决痛点 #5）

**目标**: 所有用户获得会话和上传清理。

**改动**:
1. 更新 `cleanup_old_sessions()` 接受用户列表或扫描 `data/users/` 下所有用户
2. 添加上传清理：删除超过 N 天的上传
3. 添加输出清理：删除超过 N 天的输出（或会话删除时）
4. 定期运行清理（如每小时）或会话创建时

**风险**: 低。清理是附加且幂等的。

#### 阶段 5：统一会话数据（解决痛点 #6）

**目标**: 会话元数据的单一数据源。

**改动**:
1. 确定规范位置：`.session-index.json` 成为数据源
2. 移除 `claude-data/sessions/` 中的空 JSONL 文件
3. 仅保留 `.meta.json` 用于扩展元数据（如果需要）
4. 通过扫描和填充索引迁移现有数据

**风险**: 中。需要确保合并过程中不丢失数据。

## 5. 何时重新评估

当满足以下**任一**条件时重新评估：

| 触发条件 | 指标 | 行动 |
|---------|--------|------|
| 用户增长 | >50 活跃用户 | 迁移到方案 B (SQLite) |
| 会话增长 | 每用户 >500 会话 | 迁移到方案 B |
| 消息量 | 每会话 >10K 消息 | 迁移到方案 B |
| 并发问题 | 检测到丢失更新 | 立即迁移内存到方案 B |
| 多进程部署 | 需要水平扩展 | 迁移到方案 C (PostgreSQL) |
| 查询复杂度 | 需要跨会话搜索/分析 | 迁移到方案 B 或 C |

### SQLite 迁移的便利

方案 A 的改动会使后续 SQLite 迁移更容易：
- 会话索引模式直接映射到 `sessions` 表
- 压缩的会话数据映射到 `messages` 表
- 加锁的内存文件映射到 `user_memory` 表
- 审计日志、上传和技能文件保持文件

## 6. 风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| 会话索引文件损坏 | 低 | 中 | 索引可从磁盘重建；添加写时备份（写临时文件，然后重命名） |
| 文件锁死锁 | 低 | 中 | 使用带重试的超时锁 |
| 压缩丢失数据 | 低 | 高 | 先写新文件，然后原子重命名 |
| 清理删除活跃数据 | 低 | 高 | 仅删除超过阈值且有安全边际的文件 |
| 合并期间数据丢失 | 中 | 高 | 运行前备份；运行后验证 |
| 过度工程（过早引入数据库） | 中 | 中 | 遵循上述触发指标 |

---

## 7. 总结

**当前状态**: 基于文件的存储在 2 用户 / 5MB 时工作良好。痛点确实存在但无需数据库即可修复。

**推荐**: 实施方案 A（5 个阶段的针对性修复）。这解决了所有 8 个已识别的痛点，零新基础设施，约 1 周工作量。

**未来路径**: 当系统达到 50+ 用户或每用户 500+ 会话时，迁移到 SQLite（方案 B）用于结构化数据。保持文件用于审计日志、上传和技能内容。仅当系统变成需要水平扩展的多租户时才考虑 PostgreSQL（方案 C）。
